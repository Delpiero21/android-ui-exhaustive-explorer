Problem

왜 Accessibility 하나로는 부족한가

a11y는 표준 View 계층 + 명시 시맨틱만 본다. 픽셀로 그려지거나 시맨틱이 빠진 영역은 전부 사각지대다.

01

Canvas 렌더링

View 하나로 등록되고 내부는 픽셀로만 그려짐 → 자식 노드 없음. 노트앱 손글씨가 대표 사례.

Samsung Notes 차트 손글씨
02

OpenGL / GLSurfaceView

GPU가 frame buffer에 직접 그림 → View 계층 밖. a11y tree에 흔적조차 없음.

게임 3D 뷰어 카메라 프리뷰
03

커스텀 Popup · Toast

Window 등록 방식 불일치, 짧은 표시 → AccessibilityEvent를 놓치기 쉽다.

시스템 다이얼로그 Snackbar 컨텍스트 메뉴
Approach

5-Tier 계층 결합 전략

각 Tier가 상위 Tier의 사각지대를 메운다. 단일 기법 의존을 피하고 상호 검증으로 false positive를 차단한다.

Tier 5 VLM (Vision Language Model)

사내 Ollama 한정 호출. 추상 UI · 게임 메뉴 등 4번 Tier로도 못 풀리는 화면에서만 fallback.

stuck fallback
Tier 4 Differential Probe

탭 전후 화면 diff로 실제 인터랙티브 영역만 채택. OpenGL/Canvas hit 검증의 핵심.

verify · filter
Tier 3 CV + OCR

OpenCV로 버튼·아이콘 후보 검출, OCR로 위험 단어 매칭(블랙리스트).

propose · label
Tier 2 Pixel Grid Sampling

a11y 미커버 영역을 coarse-to-fine 그리드로 스캔. Differential probe와 결합.

blind spot
Tier 1 Accessibility (a11y)

기본 클릭 후보 추출. 현재 도구의 출발점이지만 단독으로는 사각지대가 큼.

baseline
원칙 1 · 계층화
Layering
하위 Tier로 갈수록 정밀, 상위로 갈수록 광범위. 화면별로 자동 선택.
원칙 2 · 상호 검증
Cross-Validation
a11y + CV + VLM 후보의 교집합·IoU로 false positive 자동 차단.
원칙 3 · 위험 차단
Guard
결제·전송·삭제 키워드는 진입 전 블랙리스트로 차단 — 야간 자동 실행의 전제.
Coverage

기법 × UI 영역 커버리지

행: UI 영역, 열: 적용 기법. 색 강도가 높을수록 해당 기법이 그 영역을 잘 커버한다는 의미.

Roadmap

로드맵

현재 Phase 0 (Bootstrap). 다음은 Phase 1 (Baseline Explorer) → Phase 2 (Blind Spot 보완) → Phase 3 (시스템 연계).

  1. Phase 0 · Bootstrap
    레포 셋업, 문서 정립, Android/Server/Web 스캐폴딩 — 진행 중
  2. Phase 1 · Baseline Explorer
    Tier 1 (a11y) + Tier 4 (Differential Probe) 기본 DFS 엔진
  3. Phase 2 · Blind Spot 보완
    Tier 2 (Pixel Grid) + Tier 3 (CV + OCR) + Tier 5 (VLM hybrid)
  4. Phase 3 · System 연계
    dumpsys / logcat 보강 + 탐색 ↔ 추적 정합성 검증
